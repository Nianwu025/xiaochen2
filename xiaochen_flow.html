被动回复流水线 用户消息 → 小琛回复

你发一条飞书消息，小琛内部如何运转到给出回复

1 消息到达

你在飞书上发了一条消息，就像寄出一封信。这封信到达小琛的"收发室"（飞书通道），被拆开、登记，然后放进内部的传送带上。

飞书 WebSocket → 子线程接收 → asyncio.run_coroutine_threadsafe → InboundMessage
技术细节
  • FeishuChannel._on_message 在 lark-oapi 的子线程中触发
  • 通过 asyncio.run_coroutine_threadsafe 桥接到主事件循环
  • 消息被封装为 InboundMessage(chat_id, text, images) 数据类
飞书 WebSocket → 子线程接收 → asyncio.run_coroutine_threadsafe → InboundMessage
代码路径
  • infra/channels/feishu_channel.py — FeishuChannel._on_message() 在 lark-oapi SDK 的 WebSocket 回调线程中触发
  • 跨线程桥接：asyncio.run_coroutine_threadsafe(self._bus.publish_inbound(msg), self._loop)
  • InboundMessage dataclass：chat_id: str, text: str, images: list[str], reply_to: str | None
  • 飞书 WebSocket 使用长连接 + 看门狗重连机制
2 事件分发

信到了分拣中心。分拣员（EventBus）看了一眼地址，把它分到"用户来信"这条流水线上，然后通知处理员（CoreRunner）开始工作。

EventBus.emit(TurnStarted) → CoreRunner.process() → 路由分流
技术细节
  • EventBus 的 emit 模式：顺序拦截链，handler 可修改事件
  • CoreRunner 判断消息类型：内部事件 vs 普通用户消息
  • 普通消息进入 AgentCore.process() 主编排流程
EventBus.emit(TurnStarted) → CoreRunner.process() → 路由分流
代码路径
  • bus/event_bus.py — emit() 顺序拦截链，每个 handler 可修改或拦截事件
  • 8种生命周期事件：TurnStarted, BeforeReasoning, BeforeDispatch, ToolCallStarted, ToolCallCompleted, StreamDeltaReady, TurnCompleted, TurnCommitted
  • agent/core/runner.py — CoreRunner.process() 分流：内部事件走 _handle_internal()，普通消息走 AgentCore.process()
  • 四种调度模式：emit(拦截链) / observe(fire-and-forget) / fanout(并发广播) / enqueue(后台队列)

ProactiveLoop 后台心跳驱动，内容源为空时自动降级到 Drift 模式

1 心跳检测

小琛不是一直等着你说话的。他会定期"醒来"看看有没有事要做——就像保安每隔几分钟巡逻一次。你越活跃，他巡逻越勤（最快1分钟一次）；你越安静，他巡逻间隔越长（最慢8分钟一次）。

ProactiveLoop.run() → _run_loop() → AgentTick.tick()
自适应间隔 (energy三维评分)

  • D_content(0.40) — 新条目越多越高
  • D_recent(0.20) — 近期消息越多越高
  • 间隔：60s(高活跃) → 120s → 240s → 480s(低活跃)，每级±20%抖动
ProactiveLoop.run() → _run_loop() → AgentTick.tick()

  • D_energy = 1 - energy（权重0.35）：越久没说话越高
  • D_content（权重0.40）：DataGateway新条目越多越高
  • D_recent（权重0.20）：近期消息频率
  • proactive_v2/presets.py — "daily"预设：next_tick_from_score() 四档间隔

2 门控检查

巡逻之前要过四道门：①有确定的联系地址吗？②你现在忙不忙？③上次联系是不是太近了（1小时内不重复打扰）？④随机概率——不是每次巡逻都会打扰你，有20%~82%的概率决定"这次就算了"。任何一道门没过，就安静退出。

chat_id 检查 → busy veto → 1h冷却 → AnyAction概率门
AnyActionGate 概率公式
  • p = 0.20 + 0.62 × idle_factor × time_factor
  • idle_factor = 1 - exp(-idle_min / 30) — 空闲30分钟后接近饱和
  • 每日上限48次，最小间隔180秒，每天中午12:00重置
  • 任何一步失败 → 直接 return None，本轮不触发
chat_id 检查 → busy veto → 1h冷却 → AnyAction概率门

  • ① default_chat_id 是否为空
  • ② passive_busy_fn → 系统级 veto（被动回复正在处理时禁止主动触达）
